PresenceHub

Real-time presence awareness for CSR workspaces


Node.js + Express Server-Sent Events Redis Pub/Sub React Docker

The Problem

What We Are Solving

  • CSRs work on vehicle units called lots in the existing UI app
  • Multiple CSRs can open the same lot simultaneously with no awareness of each other
  • No indication on the home page when a lot is already being worked on
  • No indication on the lot detail page of who else is currently viewing it
  • Leads to duplicate work, wasted effort, and coordination gaps
  • Need a solution that works across multiple UI apps in the company — not just this one

Presence is informational only — any CSR can still open and work on any lot. This is a coordination aid, not a lock.

Solution — PresenceHub

Standalone Microservice

Built in Node.js — decoupled from the existing Java backend. Can be reused by any UI app in the company.

Server-Sent Events

Server pushes presence updates to all connected browsers in real time over a persistent HTTP connection.

Redis Pub/Sub

Shared message bus across multiple server instances. Any instance can publish, all instances receive and fan out.


React UI (existing) | SSE — receive updates (persistent connection) | REST — send actions (enter / leave / heartbeat) ▼ PresenceHub Microservice (Node.js + Express) | State + Pub/Sub ▼ Redis (bundled, swappable to central Redis later)

What Are Server-Sent Events?

SSE is a browser technology that lets a server push data to a client over a single long-lived HTTP connection. The client opens it once — the server sends messages anytime.

Browser Server | | |----GET /presence/stream--------->| (connection stays open forever) | | |<---data: presence_update---------| server pushes whenever state changes |<---data: presence_update---------| |<---: keep-alive every 30s--------| (prevents proxy timeouts) | | | connection drops (network blip) | |----reconnects automatically----->| EventSource built-in reconnect

SSE vs WebSockets vs Polling

Feature SSE ✅ WebSockets Polling
DirectionServer → ClientBi-directionalClient → Server
ProtocolPlain HTTPSeparate WS protocolPlain HTTP
Auto-reconnectYes — built inNo — manualN/A
ComplexityLowMedium–HighLow
Proxy / FirewallWorks everywhereCan be blockedWorks everywhere
OverheadLowLowHigh (repeated requests)
HTTP/2 supportYes — multiplexedNoYes

SSE is the right fit here because presence data only flows server → client. Client actions (enter/leave) are infrequent REST calls — no need for bi-directional WebSockets.

Two Connections, Not One

SSE feels bi-directional but it's actually two separate connections stitched together via Redis.

Browser PresenceHub Server Other Browsers | | | |---GET /presence/stream-->| SSE — stays open forever | | | | |---POST /presence/enter-->| | | writes to Redis | | redis.publish() | | | | | Redis subscriber fires on all instances | | | | |<--SSE presence_update----|---SSE presence_update------->| | | |

Connection 1 — SSE

Long-lived. Server → Client. Stays open. Delivers real-time updates.

Connection 2 — REST

Short-lived. Client → Server. Opens, sends data, closes. Used for enter / leave / heartbeat.

Flow: User Enters a Lot

User 1 (Tab) PresenceHub Redis User 2 (Tab) | | | | | Opens Lot #1005 | | | | | | | |--POST /enter------->| | | | { lotId, | | | | userEmail } | | | | |--HSET lot:1005---->| | | | { email, | | | | lastSeen } | | | |--PUBLISH--------->| | | | { lotId, users } | | | |<--subscriber------| | | | | | |<--SSE update--------|---SSE update----------------------->| | | | Badge appears on Lot #1005 |<--200 OK------------| | |

Flow: User Leaves Gracefully

Navigates away, clicks back, or closes tab — beforeunload fires sendBeacon

User 1 (Tab) PresenceHub Redis User 2 (Tab) | | | | | Navigates away | | | | component unmounts | | | | | | | |--POST /leave------->| | | | { lotId, | | | | userEmail } | | | | |--HDEL lot:1005---->| | | | (removes session) | | | |--PUBLISH--------->| | | | { lotId, | | | | users: [] } | | | |<--subscriber------| | | | | | |<--SSE update--------|---SSE update----------------------->| | | | Badge removed from Lot #1005 | | | Near-zero delay

Flow: Ungraceful Exit + Heartbeat

Tab Crash / Network Drop

User 1 PresenceHub Redis | | | | Tab crashes | | | x x x x | | | | | | (heartbeats stop) | | | | | [3 min] | | | [6 min] ... | | | | | | [15 min — cleanup job] | | |--HGETALL------>| | | lastSeen | | | > 15 min? | | | YES → HDEL | | |--PUBLISH------>| | |<--subscriber--| | | | | SSE update sent to all clients

Heartbeat Keep-Alive

User 1 PresenceHub Redis | | | | Tab visible | | | | | | [every 3 min] | | |--POST /-------->| | | heartbeat |--HSET--------->| | | updates lastSeen| |<--200 OK------| | | | | | Tab hidden | | | heartbeats stop | | | lastSeen freezes | | cleanup fires | | after 15 min | | | | | Tab comes back | | |--POST /enter-->| | | re-registered | |

Why Redis? Multi-Instance Problem

Multiple Node.js server instances can't share in-memory state. Redis is the shared message bus.

User A Instance 1 Redis Instance 2 User B / C | | | | | |--POST /enter------>| | | | | writes to Redis | | redis.publish() | | | | | | | |<--pub/sub---------|--pub/sub-------->| | | | | | | |<--SSE update------| | |---SSE update------>| | (clients on instance 1) (clients on instance 2)

Each instance holds its own in-memory Set of SSE connections. Redis delivers the publish to all instances. Each instance fans out to its own clients. Every tab — on any instance — gets the update.

Frontend Integration

No new React app needed. Three files to drop into the existing app.

usePresence.js

Hook — manages enter/leave/heartbeat/SSE. Handles Page Visibility API and sendBeacon on close.

LotPresenceBadge

Home page — shows avatar initials + "N viewing" next to each lot. Updates live via SSE.

LotPresenceBanner

Lot detail page — "Also viewing: a@x.com, b@x.com" banner. Auto-registers presence on mount.

// App root — one SSE connection for the whole app createSSEConnection(({ lotId, users }) => setPresenceMap(...)); // Home page <LotPresenceBadge users={presenceMap[lot.id] ?? []} /> // Lot detail page <LotPresenceBanner lotId={lotId} userEmail={currentUserEmail} presenceMap={presenceMap} />

Summary

What We Built

  • Standalone Node.js presence microservice
  • SSE + REST hybrid — simple and reliable
  • Redis pub/sub for multi-instance support
  • React POC UI to demo the full flow
  • Docker setup — one command to run

Key Design Decisions

  • SSE over WebSockets — presence is one-way
  • 15 min stale timeout — buffer for idle tabs
  • sendBeacon — reliable on tab close
  • Page Visibility API — no ghost presence
  • correlationId — full request traceability
