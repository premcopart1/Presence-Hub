PresenceHub

Real-time presence awareness for CSR workspaces

What it does

  • Tracks which users are viewing which lots — live
  • Shows a badge on the home page lot list
  • Shows an "Also viewing" banner on lot detail pages
  • Prevents duplicate work across CSR teams
  • Informational only — no locks, no restrictions

Why standalone

  • Fully decoupled from the Java backend
  • Node.js is purpose-built for real-time / event-driven workloads
  • Any UI app in the company can plug in
  • Swap bundled Redis for central DevOps Redis via one env change

Why SSE — Not WebSockets

Factor SSE ✅ WebSockets
Data directionServer → Client only — exactly what we needBi-directional — capability we'd never use
Client actions (enter/leave)Plain REST POST — simple and statelessWould send over the socket — unnecessary complexity
Auto-reconnectBuilt into EventSource — zero code neededManual — you write reconnect + backoff logic
Proxy / firewall riskPlain HTTP — works in every enterprise networkProtocol upgrade (ws://) — can be blocked
Scaling with RedisClean — each instance holds its own SSE SetSame pattern but adds protocol-level overhead
Debugging / loggingText-based HTTP — readable in any toolBinary framing — requires specialised tooling

The only real trade-off with SSE: ungraceful exits (crash / network drop) rely on a heartbeat timeout. That's an acceptable trade-off for an informational-only indicator — no one is blocked from opening a lot.

How Enter Works

Triggered when a user opens a lot detail page — or returns to a backgrounded tab

User 1 (Tab) PresenceHub Server Redis User 2 (Tab) | | | | | Opens Lot #1005 | | | | component mounts | | | | | | | |--POST /presence/enter->| | | | { lotId, userEmail } | | | | |--HSET presence:lot:1005->| | | | { correlationId: | | | | { userEmail, | | | | lastSeen } } | | | | | | | |--PUBLISH presence:updates| | | | { lotId: "1005", | | | | users: [user1] } | | | | | | | |<--Redis subscriber fires-| | | | | | |<--SSE presence_update--|----SSE presence_update----------------------->| | | | Badge appears on Lot #1005 |<--200 OK---------------| | |

The correlationId (one UUID per tab) is the Redis hash field key — so two tabs by the same user on the same lot are tracked independently.

How Leave Works

Triggered when a user navigates away from the lot detail page — component unmounts

User 1 (Tab) PresenceHub Server Redis User 2 (Tab) | | | | | Clicks back / navigates | | | | component unmounts | | | | | | | |--POST /presence/leave->| | | | { lotId, userEmail } | | | | X-Correlation-ID hdr | | | | |--HDEL presence:lot:1005->| | | | (removes this tab's entry) | | | | | | | |--PUBLISH presence:updates| | | | { lotId: "1005", | | | | users: [] } | | | | | | | |<--Redis subscriber fires-| | | | | | |<--SSE presence_update--|----SSE presence_update----------------------->| | | | Badge removed — near-zero delay

The X-Correlation-ID header is sent by the React hook on every REST call — the server uses it to find and delete exactly the right Redis entry.

How Tab Close Works

When the browser closes the tab, all HTTP connections — including the SSE stream — are terminated

User 1 (Tab) PresenceHub Server Redis User 2 (Tab) | | | | | User closes tab | | | | browser terminates all | | | | HTTP connections | | | | | | | | ~~~SSE connection closed~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ | | | | | req.on('close') fires | | | leaveByCorrelationId() | | | | | | | |--KEYS presence:lot:*---->| | | |--HGET lot:1005---------->| | | | correlationId found? | | | | YES | | | |--HDEL lot:1005---------->| | | |--PUBLISH presence:updates| | | | { lotId, users: [] } | | | | | | | |<--Redis subscriber fires-| | | | | | | |----SSE presence_update----------------------->| | | | Badge removed — near-zero delay

The SSE disconnect is always reliable — driven entirely by the server. No sendBeacon, no CORS preflight, no browser timing issues.

Inactive Tab — How It Works & Timings

Page Visibility API

When the user switches away from the tab, visibilitychange fires. The heartbeat timer is paused. When they return, POST /enter re-registers them immediately and the heartbeat restarts.

Stale Cleanup Job

A background job scans all Redis presence keys every 5 minutes. Any entry whose lastSeen is older than 15 minutes is deleted and a presence_update is published.

Summary

The Four Scenarios

1

Enter — component mounts, POST /enter → Redis HSET → publish → all SSE clients update

2

Leave — component unmounts, POST /leave → Redis HDEL → publish → all SSE clients update

3

Tab close — SSE connection drops → server detects, scans Redis by correlationId, removes, publishes

4

Inactive tab — heartbeat pauses, stale cleanup removes entry after 15 min of no activity

Key Design Decisions

  • SSE over WebSockets — presence is one-way, HTTP works everywhere
  • REST for client actions — stateless, simple, no bi-directional socket needed
  • SSE disconnect drives tab-close leave — no reliance on browser beacon APIs
  • Page Visibility API — stops heartbeat on background tab, no ghost presence
  • Redis pub/sub — works across multiple server instances out of the box
  • correlationId per tab — tracks each tab independently, full traceability
  • All timings are env vars — tune without code changes
